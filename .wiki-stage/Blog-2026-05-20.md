## Performance

### Pre-warm chunks + template caches — bootstrap freeze cut ~60% (#257)

**v0.208.0 · merged 19 May 2026**

First-join startup lag is significantly reduced by two complementary pre-warming passes that run before the eager-fill spawn loop fires.

**Chunk pre-warming.** Eager-fill calls `TrainAssembler.spawnGroup` up to ~12 times when a player first joins. Each call executes ~1 800 `setBlock` / `getBlockState` ops inside `clearSubLevelVolume` + `CarriagePlacer.placeAt`. On fresh chunks every one of those ops triggered a synchronous world-gen wait — the same pathology `TrackGenerator` already flagged as "500 ms spikes when flying over freshly-streaming chunks". Now every chunk in the planned eager-fill footprint is fetched once via `level.getChunk(cx, cz, ChunkStatus.FULL, true)` before the spawn loop begins, matching the 8 existing pre-warm sites in `PlayerJoinEvents` / `TrainBootstrapEvents`.

**Template cache pre-warming.** `CarriagePlacer.placeAt → stampBase → CarriageTemplateStore.get` read each variant's NBT from disk on first access (then cache). With 14+ groups spanning ~6 distinct variants and their parts, the first spawn touching each new variant paid a 5–2 500 ms disk-read hit on the server thread — observed as wild `place=` variance in the spawnGroup timing log. Three caches — `CarriageTemplateStore` body, `CarriagePartTemplateStore` parts, `CarriagePlacer` half-flatbed pad — are now warmed once at `ServerStartedEvent`, just before `TrainBootstrapEvents` spawns the seed train. Every subsequent placement is a cache hit, collapsing variance into a single **"Loading carriage templates…"** phase visible on the loading screen.

Per-phase timing (`clear / place / assemble / contents / total` ms) is now logged on the existing `spawnGroup` INFO line for ongoing benchmarking.

**Files changed:** `TrainBootstrapEvents.java`, `CarriagePlacer.java`, `TrainAssembler.java`, `TrainCarriageAppender.java` (+110 lines)
